## 1. 这是什么

CoworkAny 是一个桌面 AI 工作台。

它不是单纯的聊天窗口，也不是只会跑一次命令的 CLI Agent。它把下面这些能力组合在一起：

- 聊天与任务时间线
- 本地文件和命令执行
- 浏览器自动化
- Skills 与 MCP 扩展
- 长短期记忆
- 用户可见的审批与确认
- 任务中断后的恢复

如果你希望 AI 能持续参与“真实工作”而不是只回答问题，CoworkAny 的定位就是这个方向。

## 2. 快速开始

### 2.1 环境要求

- `Node.js >= 22.22.1 < 23`
- `Bun >= 1.2.0`
- `Rust` 与 Tauri 开发环境
- `Python 3.x`

### 2.2 安装依赖

```bash
cd desktop
npm install

cd ../sidecar
bun install
```

### 2.3 配置模型

创建 `sidecar/llm-config.json`：

```json
{
  "provider": "anthropic",
  "anthropic": {
    "apiKey": "sk-ant-...",
    "model": "claude-sonnet-4-5"
  }
}
```

如果你需要天气、新闻、Google Calendar 或 Gmail 相关能力，可参考根目录的 [.env.example](../.env.example) 配置额外环境变量。

### 2.4 启动应用

```bash
cd desktop
npm run tauri dev
```

如果你只想单独调试某个后端服务，也可以分别启动：

```bash
cd sidecar
bun run src/main.ts
```

```bash
cd rag-service
python main.py
```

```bash
cd browser-use-service
python main.py
```

## 3. 核心概念

### 3.1 Workspace

Workspace 是 CoworkAny 的工作上下文边界。它通常对应一个项目目录或一类任务目录。

一个工作空间通常会关联：

- 当前项目路径
- 会话与任务上下文
- Skills / MCP 配置
- 包管理器识别结果
- 局部偏好和记忆

### 3.2 Task

Task 是一次实际工作执行单元。它不只是“一条消息”，而是从输入、规划、工具调用、审批、结果返回，到可能的恢复点持久化的一整个生命周期。

### 3.3 Skill

Skill 是可复用的任务规则或流程提示，通常用于：

- 编码规范
- 某类任务的执行方式
- 特定领域操作步骤
- 高层约束

### 3.4 MCP

MCP（Model Context Protocol）是 CoworkAny 接入外部工具和服务的统一入口。

如果你希望 AI 使用现有内部服务、文件系统服务器、数据库桥接或第三方工具包，通常会落到 MCP 这一层。

### 3.5 Approval

当任务涉及高风险副作用时，CoworkAny 会触发审批或确认，例如：

- 写文件
- 执行 Shell
- 发起外部网络请求
- 进行高风险浏览器操作

这不是多余步骤，而是系统安全模型的一部分。

### 3.6 Memory

CoworkAny 不是“每次重新开始”的工具。它会在不同层级上保留上下文：

- 当前任务上下文
- 会话级历史
- 长期知识与偏好
- 可检索的 Vault 内容

## 4. 第一次使用的推荐路径

如果你是第一次打开 CoworkAny，建议按这个顺序操作。

### 4.1 配置模型

在设置页中选择模型 Provider，填入 API Key，并保存。

建议先用一个你最熟悉、最稳定的 Provider 开始，不要一上来同时配置多个。

### 4.2 创建或选择工作空间

把当前项目或任务目录设置为 Workspace。

这样做的好处是：

- AI 有明确的工作上下文
- 项目检测更准确
- Skills 与 MCP 更容易关联
- 后续恢复与记忆更有边界

### 4.3 从一个简单任务开始

第一条消息建议不要太大，先让系统建立上下文，例如：

```text
帮我概览这个仓库的结构，并告诉我从哪里启动开发环境
```

或者：

```text
先不要改代码，分析一下这个项目的前后端分层
```

### 4.4 再发起需要执行的任务

当你已经确认上下文正常，再进入真正执行任务，例如：

```text
修复这个页面的状态同步问题，并补一条测试
```

```text
帮我打开目标站点并抓取正文摘要
```

## 5. 常见使用方式

### 5.1 代码分析

```text
分析 sidecar 的任务恢复流程，给我一个从入口到持久化的调用链
```

### 5.2 修复问题

```text
修复 desktop 中断任务恢复后的 UI 状态不同步问题，并运行相关测试
```

### 5.3 代码审查

```text
审查这组改动，优先找行为回归、缺少验证和高风险边界问题
```

### 5.4 浏览器自动化

```text
打开目标网站，登录后检查表单提交链路是否有明显报错
```

### 5.5 知识沉淀

```text
把这次排障过程整理成一条可复用技能，并说明它适合什么场景
```

## 6. 工作空间

### 6.1 为什么工作空间重要

很多 Agent 工具之所以容易“跑偏”，是因为它们没有清晰的上下文边界。

CoworkAny 通过 Workspace 把下面这些内容绑在一起：

- 当前项目目录
- 会话上下文
- 项目技术栈线索
- Skills / MCP 选择
- 包管理器识别

### 6.2 适合怎样组织工作空间

推荐把一个工作空间对应到：

- 一个代码仓库
- 一个文档项目
- 一个长期任务目录

不建议把多个完全无关的项目混在同一个 Workspace 里。

### 6.3 关于包管理器识别

CoworkAny 会根据项目特征识别包管理器，例如：

- `package.json` 中的 `packageManager`
- 锁文件
- 项目配置

这会影响后续命令建议与执行行为。

## 7. Skills 与 MCP

### 7.1 什么时候用 Skill

当你希望某类任务“每次都按类似方式执行”时，就应该考虑 Skill，例如：

- 前端组件开发规范
- 代码审查习惯
- 调试流程
- 文档产出格式

### 7.2 什么时候用 MCP

当你希望接入“系统本来没有内建的外部能力”时，就应该考虑 MCP，例如：

- 文件系统服务器
- 数据库工具
- 外部 API
- 企业内部服务

### 7.3 组合使用的价值

Skill 决定“怎么做”，MCP 决定“能做什么”。

把两者组合起来，CoworkAny 才能既有执行边界，又有领域扩展能力。

## 8. 审批与安全

### 8.1 为什么会弹确认框

因为 CoworkAny 不是只读助手。它会尝试真正参与执行，而执行就意味着风险。

确认框通常出现于：

- 需要写入文件
- 需要执行命令
- 需要访问外部网络
- 需要进行高风险工具操作

### 8.2 怎样理解审批

审批不是“系统卡你一下”，而是把以下问题显式化：

- 这个动作是否真的应该发生
- 它会影响哪里
- 风险是否值得承担

### 8.3 实际使用建议

- 初次运行某类任务时，先看清楚它要改什么
- 大范围写入或批量操作时，不要盲目快速通过
- 如果你还在探索上下文，先要求“只分析，不修改”

## 9. 记忆与恢复

### 9.1 记忆有什么用

记忆让 CoworkAny 不必每次都重新理解你：

- 偏好的技术栈
- 之前的任务背景
- 历史讨论过的问题
- 可沉淀成知识的经验

### 9.2 中断恢复意味着什么

如果任务在执行过程中被中断，CoworkAny 会尝试恢复运行时状态，并在 UI 中继续呈现任务上下文。

这让它更像一个长期协作者，而不是单次对话工具。

### 9.3 用户如何用好这点

- 不要每次都开启全新、毫无上下文的任务
- 长任务尽量围绕同一个 Workspace 持续推进
- 如果你想利用连续性，就明确引用前一个任务或问题

## 10. 常见问题

### 10.1 应用启动了，但 AI 不工作

先检查：

- `sidecar/llm-config.json` 是否存在且格式正确
- API Key 是否有效
- Provider 是否可用

### 10.2 为什么会提示审批

因为当前任务触发了高风险副作用。最常见的是写文件、执行命令或网络访问。

### 10.3 为什么它有时不直接改代码

常见原因有：

- 当前上下文不足
- 风险太高，需要确认
- 任务更适合先分析再执行
- 工作空间边界不清晰

### 10.4 为什么建议先建 Workspace 再发任务

因为没有 Workspace，系统就更难准确判断：

- 该看哪些文件
- 该用什么包管理器
- 哪些上下文值得保留

### 10.5 我该怎么写 prompt

优先写清楚这几件事：

- 目标是什么
- 是否允许改代码
- 是否要运行测试
- 是否只要分析

例如：

```text
先不要改代码，分析这个仓库的启动链路，并列出关键入口文件。
```

```text
修复这个问题，可以改代码并运行相关测试，最后告诉我验证结果。
```
